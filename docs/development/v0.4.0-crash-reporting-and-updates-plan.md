# v0.4.0 — Crash Reporting and Updates: Wave Plan

Status: planning. This is the implementation map for the 0.4.0 wave. Authoritative architecture
lives in **ADR 0012** (update security model, with the 0.4.0 clarification) and **ADR 0017**
(crash reporting). This document only sequences the work and records the gates.

## Goal

Ship two independent strands that move ExoSnap toward a reliable `1.0`:

1. **Crash reporting** — capture native crashes locally, present a single scrubbed report dialog
   (immediate + next-launch), assisted manual delivery via GitHub issue (Stage 0).
2. **Updates** — check for new versions against GitHub Releases, Stable/Preview channels,
   signed-manifest + hash verification, installed-mode auto-update, rollback/downgrade protection.

## Infrastructure stance (why this wave needs no ExoSnap server)

| Capability | How it works | External dependency |
|---|---|---|
| Update check / channels | Public GitHub Releases API; Stable = latest, Preview = prerelease | none |
| Update integrity | ed25519 manifest signature; key in CI secret; public key embedded in client | none |
| Update hosting | GitHub Release assets | none |
| Authenticode signing | Optional late-bound CI step | **pending SignPath cert** (not a blocker) |
| Crash capture / dialog / scrub | All local, offline | none |
| Crash delivery (Stage 0) | Prefilled GitHub issue + manual `.dmp` attach + clipboard | none |
| Crash delivery (Stage 1) | Automated minidump upload (e.g. Sentry) | **upload target + privacy review** (gated, deferred) |
| Symbol pipeline | PDBs archived as CI/release artifacts; offline symbolication | none |

The only externally-blocked pieces are the **automated** crash upload and **Authenticode**
signing. Both sit behind gates and do not block the rest of the wave.

## Slices

### Crash strand

**Crash A — Crashpad integration + local capture**
- Vendor Crashpad via FetchContent; vet its license against the project's GPL model *before*
  wiring (mirror the x264 gate in ADR 0007).
- Out-of-process handler; minidump + structured metadata to `%LOCALAPPDATA%\ExoSnap\crashes\`.
- Package the handler executable in the portable ZIP and the MSI; resolve it at runtime via the
  application directory.
- Coordinate with the recovery manifest (ADR 0015): the crash report references recovery state,
  does not duplicate it.
- Tests: a simulated crash produces a dump at the expected path; handler is found at runtime.

**Crash B — Report dialog + scrubbing + next-launch detection**
- On startup, detect an unhandled dump; build the symbolicated text report.
- Privacy scrubbing via **allowlist** (OS build, GPU/driver, app version, encoder backend,
  container/codec); strip usernames/paths/machine name/recording filenames.
- Single-stage dialog: scrubbed report, collapsed-by-default details; two-column
  "What gets sent / Never sent" transparency block; actions Restart / Report on GitHub /
  Open crash folder / Close.
- Update the design mappe crash mock (`.workspace/design/mappe-bundle/.../mappe-waves-late.jsx`)
  to the shipping state before building: keep the transparency columns, drop the "anonymous
  install id", replace the Stage-1 "Send report" primary with the Stage-0 GitHub-issue flow,
  correct the "never mid-capture" note (immediate reporter + next-launch).
- Stage 0 delivery: `.github/ISSUE_TEMPLATE/crash.*` template; prefill URL with stack + allowlisted
  metadata; instruct manual `.dmp` attach; clipboard fallback for users without a GitHub account.
- Ordering: recovery overlay first, crash dialog second; no double-prompt.
- Tests: scrubbing removes paths/username; recovery-then-crash ordering; prefill URL well-formed
  and within length budget.

**Crash A2 — immediate reporter process**
- Lightweight separate reporter launched after the dump is written; shows the Crash B dialog at
  once; `FlashWindowEx`, no hard focus theft.
- "Recording secured" reassurance when a session was active.
- Mark a dump handled by the reporter so the next-launch path does not re-prompt.
- Tests: reporter launches and renders; flash invoked; dump de-dup across both paths.

**Crash C — symbol pipeline + gated automated upload** *(partially external-blocked)*
- CI/release archives PDBs as artifacts/debug assets, keyed by build id + version.
- Optional automated upload behind `EXOSNAP_OFFICIAL_BUILD` + explicit opt-in; compiled out /
  hard-disabled until a concrete target exists, passes privacy review, **and the hosting network
  is hardened**. Preferred target: self-hosted, Germany-only endpoint (maintainer VPS/homelab) —
  verify native Crashpad minidump ingest; self-hosted Sentry is RAM-heavy (won't fit a small VPS).
  No persistent install id; at most a per-report random correlation id.
- Update `PRIVACY.md` with the concrete data set and target when upload actually ships.

### Update strand

**Update A — update check + channels + settings + guards**
- Version compare against the public GitHub Releases API (no token).
- Stable/Preview channel model (latest vs prerelease); settings UI (channel select, check-on-start
  opt-in).
- Guard: no update check/download/install during an active recording or finalization.
- Portable vs installed detection; portable = notify-only.
- `EXOSNAP_OFFICIAL_BUILD` gate; self-built = update check compiled out.
- Tests: version compare; channel selection; recording guard; portable detection; self-built gate.

**Update B — manifest format + signature/hash verification + CI signing**
- Manifest schema (version, minimum-accepted-version, package URLs, SHA-256, ed25519 signature).
- Client verifies manifest signature (embedded public key) before reading content; verifies SHA-256
  of the downloaded package before executing.
- CI release step signs the manifest with the ed25519 key from a CI secret.
- Tests: valid signature accepted; tampered manifest/package rejected; downgrade below
  minimum-accepted-version rejected; no partial binary retained on failure.

**Update C — installed auto-updater + rollback** *(Authenticode gate, not a blocker)*
- Installed mode: download, verify, hand off to installer; **no silent restart** (user initiates);
  pending update applied on next launch only after confirmation.
- Rollback/downgrade with user confirmation, permitted only above minimum-accepted-version.
- Authenticode signing of installer/package = optional late-bound CI step (awaits SignPath).
- Tests: updater flow (mocked); rollback rules; no-auto-restart invariant.

## Sequencing and dependencies

- Crash: **A → B → A2**, then **C** (C is partially external-blocked; can trail).
- Update: **A → B → C** (C's Authenticode step is optional/deferred; the rest of C ships without it).
- The two strands are independent and may interleave. Per AGENTS.md, do not let parallel workers
  edit shared high-churn files (`MainWindow.cpp`, `ConfigPage.cpp`, shared CMake) concurrently.

## Out of scope for 0.4.0 (explicit)

- Automated crash upload to a live backend (Stage 1) — until a target + privacy review exist.
- Authenticode code-signing / SmartScreen reputation — awaits the SignPath certificate; broader
  installer/reputation work stays a later-version concern per the roadmap.
- Any ExoSnap-operated server, custom symbol server, or client-side API token.
